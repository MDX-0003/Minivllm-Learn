# Skills — `agent.md` Todo 写作规范

## 文档定位

本文件用于约定：

> 以后每次需要把任务拆解、实施计划、阶段性 todo 写入 `agent.md` 时，应该遵守什么格式、粒度、边界与质量要求。

它不是业务实施文档，不直接承载某个功能的 todo 本体；它是 **`agent.md` 的写作规范**。

---

## 文件位置约定

本规范固定放在仓库根目录：

- `skills.md`

原因：

1. 与根目录的 `agent.md` 同级，最容易被后续 agent / LLM 发现。
2. 不和业务代码、插件内部文档、引擎源码文档混在一起。
3. 适合作为整个仓库层面的通用约束，而不是某个子模块私有规则。

如果后续仓库出现更多 agent 级规范文件，仍优先保留这个路径，不频繁搬迁。

---

## `agent.md` 的职责边界

`agent.md` 只应该承载两类内容：

1. **当前上下文所必须知道的真实系统说明**
	- 当前渲染链路
	- 当前关键代码位置
	- 当前瓶颈结论
	- 当前任务目标定义

2. **当前任务的可执行 todo 计划**
	- 面向后续连续实施
	- 面向 code review
	- 面向逐步编译验证

`agent.md` 不应该承担以下角色：

- 历史排查日志全集
- 所有试错记录的堆叠区
- 和当前任务无关的长期知识库
- 大段过期结论的归档仓库

如果旧内容和当前任务无关，应该删除或覆盖，而不是继续叠加。

---

## 什么时候应该重写 `agent.md`

满足以下任一条件时，允许直接重写 `agent.md`，甚至清空旧内容后重写：

1. 旧文档的结论已经和当前代码或当前 capture 不一致。
2. 旧文档主要在记录历史排查，已经不适合作为实施清单。
3. 当前任务进入了新的阶段，旧 TODO 无法继续指导实施。
4. 用户明确要求按新的风格或新的目标重新整理。

重写时保留的唯一原则是：

> 只保留“当前继续工作所必须知道”的内容。

---

## 写 `agent.md` 的总体风格

以后写入 `agent.md` 时，必须遵守以下风格：

### 1. 先给出“文档目标”

开头必须先说明：

- 这个 `agent.md` 现在是干什么用的
- 保留哪两类内容
- 为什么不保留其它内容

建议使用类似结构：

- `## 文档目标`
- `## 当前链路`
- `## 关键位置`
- `## 当前结论`
- `## 目标定义`
- `## TODO ...`

### 2. 只写经过确认的链路

链路图、pass 顺序、瓶颈结论，只能写：

- 已从代码确认的内容
- 已从 capture / profiling 确认的内容

不能写推测性的陈述冒充现状。

如果存在假设，必须明确标记为：

- “建议”
- “第一版策略”
- “待验证”

### 3. 用工程文档语气，不用聊天语气

应该：

- 直接
- 明确
- 可执行
- 可 review

不要：

- 写鼓励性口号
- 写和实施无关的解释性废话
- 写“可能、也许、大概”式长段模糊表述

---

## `agent.md` 中 todo 的拆分原则

Todo 必须按“最小可编译闭环”拆分，而不是按概念粗拆。

### 核心原则

每个 TODO 必须同时满足：

1. **单独提交后可以编译**
2. **单独 code review 时边界清楚**
3. **失败时容易回滚**
4. **成功后能为下一步提供稳定地基**

### 正确的拆法

应该拆成：

- 数据结构
- 数据构建
- GPU 资源上传
- Shader 新增但未接主链
- Render graph 接线
- 调试开关
- Profiling 验证

### 错误的拆法

不要出现这种 TODO：

- “把整个 LOD 做完”
- “把 shader 和 C++ 都改掉并优化性能”
- “顺手把 draw / sort / transform 都一起重构”

这类 TODO 无法独立编译验证，也无法独立 review。

---

## 每个 TODO 必须包含的字段

以后在 `agent.md` 里写 TODO 时，每一个 TODO 都必须至少包含以下小节。

### 必须有

- `## TODO N：标题`
- `### 目标`
- `### 改动文件`
- `### 需要新增的内容` 或 `### 实现要求`
- `### 本步不做的事`
- `### 编译边界`
- `### review 关注点`
- `### 验收标准`

### 可选但推荐

- `### 第一版策略`
- `### 风险点`
- `### 回滚边界`
- `### 后续依赖`

---

## 每个 TODO 的写法要求

### 1. 标题必须体现“动作 + 边界”

标题不能太泛。

推荐：

- `TODO 3：把 Chunk LOD 元数据上传到 GPU，但默认不被使用`
- `TODO 7：改造 ComputeTransformChunkAware.usf，支持按 LOD 子集展开（最小侵入版）`

不推荐：

- `TODO：做 LOD`
- `TODO：优化渲染`

### 2. `目标` 只写这一小步的完成定义

`目标` 不要写成整个项目目标。

应该说明：

- 本步完成后，系统新增了什么能力
- 但仍然没有打开哪些行为变化

### 3. `改动文件` 要尽量精确

应写到具体文件级别。

例如：

- `Plugins/PICOSplat/Source/PICOSplatRuntime/Public/SplatAsset.h`
- `Plugins/PICOSplat/Shaders/Private/ComputeTransformChunkAware.usf`

如果文件尚不确定，可以写“可能涉及”，但必须是小范围。

### 4. `本步不做的事` 必须写

这是为了控制 patch 边界。

如果不写，很容易在实施时顺手把三个阶段一起改掉。

`本步不做的事` 应明确排除：

- 不接主链
- 不切默认逻辑
- 不做性能调参
- 不合并 pass
- 不重构其它模块

### 5. `编译边界` 必须可验证

要明确说明本步结束后：

- 是否要求全工程可编译
- 默认开关关闭时是否要求零行为变化
- 是否允许只增加资源但不接执行路径

### 6. `review 关注点` 不能是空话

应该指出 reviewer 真正要重点看的点，例如：

- 生命周期是否一致
- 索引映射是否正确
- 尾 chunk 处理是否正确
- 是否引入隐式行为变化

### 7. `验收标准` 必须能落地判断

验收标准要能被明确验证，不要写成“看起来差不多”。

应该是：

- 编译通过
- 默认路径行为不变
- 新字段可序列化
- capture 中某个 dispatch 明显下降
- `VisibleCount[0]` 与 draw args 一致

---

## `agent.md` 中链路说明的写法要求

如果要写当前渲染链路或数据流，必须遵守下面的方式：

### 1. 用顺序编号

推荐：

```text
[1] ChunkCull
[2] ChunkCompact
[3] GenChunkIndirectArgs
...
```

### 2. 每一步尽量写三件事

- 输入
- 输出
- 当前关键语义

### 3. 只保留当前任务必须知道的 pass

不要为了“完整”把所有无关 pass 都抄进来。

例如当前做 Chunk LOD，就重点写：

- ChunkCull
- ChunkCompact
- Transform
- PrefixVisible
- Sort
- Draw

而不是把整个 UE frame graph 都塞进 `agent.md`。

---

## `agent.md` 中性能结论的写法要求

### 1. 只写当前最相关指标

只列出当前决策真正需要的 timing。

例如当前做 Chunk LOD，就重点列：

- `ChunkCull`
- `ChunkCompact`
- `ComputeTransforms_ChunkAware`
- `ChunkPrefixVisible`
- `Distances`
- `RadixSort`
- `IndirectDrawIndexed`

### 2. 明确写结论，不只堆数字

数字后必须有判断，例如：

1. 哪个是主瓶颈
2. 哪些 pass 已经足够快
3. 为什么当前优化应该插在这里

### 3. 不保留过期 profiling 结果

如果新的 capture 结论已经推翻旧结论，旧内容应该删除或重写，不要两个版本并列堆着。

---

## `agent.md` 中禁止出现的内容

以后写 todo 到 `agent.md` 时，禁止出现以下问题：

### 1. 过时结论和当前结论混放

如果旧结论已经不适用，应直接删，不要保留“历史版本 A / 历史版本 B / 当前版本 C”同时出现，除非当前任务明确就是做历史对比分析。

### 2. 一个 TODO 覆盖多个编译边界

禁止把下面内容放进同一个 TODO：

- 新数据结构
- 新 shader
- 新 render graph 接线
- 默认路径切换
- profiling 调参

### 3. 空泛标题

例如：

- “优化性能”
- “改善 LOD”
- “重构渲染逻辑”

### 4. 没有 `本步不做的事`

缺少这一栏的 TODO 默认视为边界不清。

### 5. 没有可验证验收标准

没有可验证收口条件的 TODO，后续无法判断是否完成。

---

## 推荐的 `agent.md` 结构模板

以后若需新开一个完全不同主题的 `agent.md`，优先使用以下结构：

```markdown
# Agent.md — [当前任务主题]

## 文档目标

## 当前链路 / 当前系统说明

## 当前关键代码位置

## 当前瓶颈 / 当前问题结论

## 目标定义

## 实施原则

## TODO 总览

## TODO 1：...
### 目标
### 改动文件
### 需要新增的内容 / 实现要求
### 本步不做的事
### 编译边界
### review 关注点
### 验收标准

## TODO 2：...
...
```

如果任务比较小，也至少保留：

- `文档目标`
- `当前结论`
- `TODO`

---

## 推荐的写作流程

以后每次要写 `agent.md` 时，建议按这个顺序来：

1. 先确认当前链路与当前结论
2. 删除旧的过期内容
3. 先写“文档目标”与“当前结论”
4. 再写当前关键链路
5. 最后按独立编译边界拆 TODO
6. 回读检查每个 TODO 是否：
	- 能独立编译
	- 能独立 review
	- 有明确验收标准

---

## 最低质量标准

只要未来写入的 `agent.md` 不满足下面任一条，就应该返工：

1. 读者无法在 2 分钟内知道当前链路和主瓶颈。
2. 某个 TODO 无法独立形成一个安全 patch。
3. 某个 TODO 没写清楚“本步不做的事”。
4. 某个 TODO 完成后是否成功无法判断。
5. 文档保留了明显过时且会误导实施的内容。

---

## 一句话原则

以后往 `agent.md` 写 todo 时，始终遵守这一条：

> `agent.md` 不是聊天记录，也不是历史堆栈；它必须是“当前阶段可直接执行的工程作战清单”。
