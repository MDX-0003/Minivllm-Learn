# MyvLLM 的 Qwen2.5-VL 教学路线图

## 文档定位

这不是一份“泛泛而谈的目标清单”。
它是一份面向当前仓库状态的、以学习为中心的路线图。

它要解决的不是：

- “最后我要不要支持 VL”
- “是不是应该直接照着官方实现写一遍”

而是更具体的问题：

- 为什么这件事值得做？
- 为什么必须先理解官方契约，再写本地代码？
- 应该按什么顺序学，才不会在实现时每一行都反复问 `why`？
- 每个阶段做到什么程度，才算真的掌握，而不是“好像看懂了”？

这份路线图默认你的最终目标是：

- 交付一个**教学型 mini framework**
- 配套一份能讲清楚 VL serving contract 的文档
- 代码为主，文档为辅
- 但代码必须建立在对官方 Qwen2.5-VL 推理契约的理解之上

---

## 为什么需要这份新路线图

现有的 [qwen2_5_vl_inference_roadmap.md](./qwen2_5_vl_inference_roadmap.md) 已经给出了正确的大方向：

- 不要再把多模态仅仅当作“文本模型外面包了一层图像胶水”
- 要转向真实的 Qwen2.5-VL checkpoint
- 要把当前框架改造成一个能够承载真实 VL 推理契约的学习型框架

但如果目标是“教学型项目”，仅有这个方向还不够。

原因很现实：

1. 如果只知道最终目标，不知道为什么先做哪一步，你会在实现中频繁停下来问 `why`，推进非常慢。
2. 如果只追求“跑起来”，很容易发明一套和官方不一致的本地协议，最后学偏。
3. 如果为了不跑偏而直接抄官方 `processor` 或模型适配逻辑，又很容易停留在“复刻”而不是“理解”。

所以，教学型路线必须同时满足三件事：

- 以官方**可观测行为**作为真值
- 用本地代码自己重建一个更小、更清楚的实现
- 在每个阶段都设置“提问式验收”，防止自己以为懂了

这正是这份文档要做的事。

---

## 当前仓库的起点

基于当前仓库，你已经不是从零开始。

你已经有：

- 一个可运行的官方基线脚本：[run_qwen2_5_vl_baseline.py](./../run_qwen2_5_vl_baseline.py)
- 一份基线检查清单：[qwen2_5_vl_baseline_checklist.md](./qwen2_5_vl_baseline_checklist.md)
- 一个正在承载“类 vLLM 推理学习”的引擎骨架：[mm_llm_engine.py](./../src/myvllm/engine/mm_llm_engine.py)

这意味着：

- 你已经证明真实 checkpoint 在本机可跑
- 你已经开始观察官方输入输出
- 你已经有一个可以承载 scheduler / prefill / decode 这些概念的本地工程骨架

但同时也要看清当前局限：

- 你还没有完整掌握官方 processor 契约
- 你还没有把“视觉 token 长度为什么是这样”彻底讲清楚
- 你还没有一个清晰的本地 `VLProcessor -> VLModelAdapter -> Engine` 职责划分
- 你还没有把“教学路径”和“真实 checkpoint 路径”分离清楚

因此，接下来的路线不能是“直接上手重写完整 VL 类”，而必须是：

- 先建立真值
- 再建立抽象
- 再写教学版实现
- 最后再把它和 mini-vLLM 引擎接起来

## 先解释几个后文会反复出现的词

### 1. 契约

这里的“契约”，不是法律意义的合同，而是“上下游之间必须共同遵守的接口约定”。

例如：

- 上游到底传 `prompt: str`，还是传 `messages`
- processor 到底返回 `input_ids`，还是同时返回 `pixel_values`、`image_grid_thw`
- model adapter 到底接收哪些字段
- engine 需要知道哪些细节，不需要知道哪些细节

如果这些约定不清楚，代码虽然也能写，但会非常容易发散和重复猜测。

### 2. Processor

在这个项目里，`Processor` 不是“tokenizer 的别名”。
它更像一个“多模态输入装配器”。

它至少要负责：

- 接收结构化输入
- 渲染 prompt
- 调用图像预处理
- 产出模型真正需要的输入

如果你把它理解成“只是多包了一层 tokenizer”，后面很多设计都会偏掉。

### 3. Model Adapter

`ModelAdapter` 的意思是：

- 站在“本地框架”和“真实模型”之间做翻译

它负责回答：

- 本地框架交给模型的是什么
- 真实 Qwen2.5-VL checkpoint 期待的是什么
- 模型输出怎么回到本地采样/解码路径里

它不是重新实现模型本身，而是把框架和模型对接起来。

### 4. Prefill / Decode

这是生成式推理里很重要的一组概念。

- `prefill`：把整段已有输入一次性送进模型，建立上下文和 KV cache
- `decode`：在已经有 KV cache 的前提下，一次只生成少量新 token

在多模态里，图像通常只参与 `prefill` 阶段。
也就是说，图像不是每一步 decode 都重新编码一遍。

### 5. `image_grid_thw`

这是 Qwen2.5-VL 中非常关键的视觉元数据。

- `t` 表示时间维度
- `h` 表示视觉网格的高
- `w` 表示视觉网格的宽

对于单张图片，通常 `t=1`。
它不是原图分辨率本身，而是图像经过预处理、切 patch、再映射到视觉网格之后的尺寸描述。

### 6. patch / merge / image token

这三个词容易混。

- `patch`：图像被切出来的小块
- `merge`：多个 patch 在视觉塔或输入协议中被合并
- `image token`：最终在 LLM 输入序列中占位的视觉 token

所以，图像 token 的个数通常不是“patch 的个数”，而是“经过 merge 之后保留下来的视觉位置数”。

### 7. 教学路径 vs 真实路径

这个项目里最好明确保留两条路径：

- 教学路径：为了帮助自己理解而构建的简化实现
- 真实路径：尽量忠实于官方 Qwen2.5-VL 契约的实现

保留两条路径没有问题。
问题只会出现在你把它们混为一谈的时候。

## 总体原则

在进入各个里程碑之前，先把下面几条原则固定下来。

### 原则 1：先对齐官方可观测行为，再讨论内部实现

不要先问：

- 官方内部到底是怎么写的？

先问：

- 官方对同一个样例，外部可观测结果是什么？

包括：

- `messages` 长什么样
- prompt text 长什么样
- `input_ids` 长什么样
- `image_grid_thw` 是多少
- `image_token_id` 重复了多少次
- 最终 completion 如何裁剪

教学项目要复制的是“行为”，不是“源码行数”。

### 原则 2：先做单图、单轮、单模型

第一版不要碰：

- 多图
- 视频
- 多轮 agent
- 高性能 batching
- 各种模型统一抽象

这些都是真问题，但不是第一阶段最值钱的问题。

第一版最值钱的问题是：

- 官方 Qwen2.5-VL 的最小推理契约到底是什么？

### 原则 3：每个阶段都要留下“可对照样本”

如果一个阶段结束后，你没有留下：

- 可复现的日志
- 小型文档
- 对照表
- 样本输入输出

那下一阶段你大概率还会重新猜一次。

### 原则 4：每个里程碑都必须能回答问题

这份路线图里，每个里程碑都会配一组问题。
如果你在阶段结束时还回答不出来，就不要急着进入下一阶段。

这不是形式主义，而是为了避免“代码在前，理解在后”的状态一直持续。

---

## 路线总览

建议按下面 7 个里程碑执行：

1. 建立官方真值样本
2. 解剖官方 processor 契约
3. 解剖视觉 token 长度与视觉 tensor 路径
4. 建立本地教学抽象与模块边界
5. 实现教学版 `VLProcessor`
6. 实现真实 checkpoint `VLModelAdapter`
7. 把真实路径接入 mini-vLLM 引擎并验证 prefill/decode

顺序不要反。
你最终要的是一个教学型 mini framework，但这个目标必须建立在前 3 个“理解阶段”已经扎实完成的前提上。

## 里程碑 1：建立官方真值样本

### 为什么先做这个

因为没有“真值样本”，后面所有 debug 都会变成猜谜。

一旦本地实现输出不对，你会无法判断问题在：

- prompt
- 图像预处理
- token 数量
- 位置编码
- engine 长度统计
- 还是模型本身

官方真值样本的作用，就是给后面所有本地工作提供对照标准。

### 这里会遇到的名词

`known-good sample`
意思是“已知正确样本”。
它不是大量测试集，而是一个你能完全看懂、完全记录、之后反复拿来对照的固定样例。

### 这一步具体要做什么

以 [run_qwen2_5_vl_baseline.py](./../run_qwen2_5_vl_baseline.py) 为起点，固定一张图片和一个问题，记录：

- 原始 `messages`
- `apply_chat_template(tokenize=False)` 的结果
- `input_ids`
- `vision_start_token_id`
- `vision_end_token_id`
- `image_token_id`
- `pixel_values.shape`
- `image_grid_thw`
- `generated_ids`
- 裁剪后的 completion
- 最终 answer text

并把这些内容写成一份稳定的记录文档。

### 这一步先不要做什么

- 不要急着改 engine
- 不要急着抽象本地 processor
- 不要急着支持第二张图或第二种模型

### 建议产出物

- 一份基线记录文档
- 一份带标签的日志脚本输出

### 关键提问

- 为什么 `messages` 才是概念上的输入，而不是单个 prompt 字符串？
- 官方的 prompt text 到底是 processor 生成的，还是用户自己拼出来的？
- 传给模型的输入里，哪些是文本侧字段，哪些是视觉侧字段？
- 为什么 `generate()` 的输出不能直接全部 decode，而要先裁掉 prompt 前缀？
- 当前样例里，哪些结果可以被当作后续所有本地实现的真值？

### 完成标准

如果你能不看源码，只看自己保存的样本记录，就清楚说出这一次官方推理完整经过了哪些阶段，这个里程碑才算完成。

## 里程碑 2：解剖官方 Processor 契约

### 为什么第二步不是写代码，而是继续分析 Processor

因为你接下来最容易写错的，不是模型，而是输入。

对多模态模型来说，“输入长什么样”不是附属问题，而是核心问题。
如果 processor 契约没有看懂，你写出的本地 `VLProcessor` 只会是一套“看起来像”的本地协议。

### 这里会遇到的名词

`chat template`
表示把结构化的对话数据渲染成模型可消费的文本模板。

`placeholder`
表示视觉内容在文本序列中的占位标记。
它的作用不是表达图像语义本身，而是给模型一个“这里有视觉内容”的结构信号。

### 这一步具体要做什么

继续基于基线脚本和 [qwen2_5_vl_baseline_checklist.md](./qwen2_5_vl_baseline_checklist.md)：

1. 把 `apply_chat_template(tokenize=False)` 和 `tokenize=True` 的路径拆开观察。
2. 明确：
   - 谁负责生成 prompt text
   - 谁负责插入图像占位符
   - 谁负责把占位符转成 token ids
3. 对比：
   - `messages`
   - 渲染后的 prompt text
   - 最终 `input_ids`

### 这一步最重要的学习任务

你要回答一个经常被忽略的问题：

- 官方 processor 究竟做了哪些“不是 tokenizer 能替代”的事情？

如果你能答出来，本地教学版 `VLProcessor` 的职责边界就会开始清楚。

### 这一步先不要做什么

- 不要照抄官方 `processor` 类
- 不要急着抽象成统一多模型接口
- 不要急着支持视频

### 建议产出物

- 一张三列对照表：
  - `messages`
  - prompt text
  - token ids / 特殊 token 区间

### 关键提问

- 图像占位符是在 `messages` 里直接写死的吗，还是由 processor 逻辑生成的？
- `tokenize=False` 和 `tokenize=True` 各自让你看到什么不同的信息？
- 为什么说 `VLProcessor` 不只是 tokenizer 的包装？
- 如果本地框架还停留在 `prompt: str + image_path: str`，它会丢掉哪些语义？
- 本地 processor 最终应该拥有哪几类职责，哪些职责不该归它？

### 完成标准

如果你能够画出一条清晰链路：

- `messages -> prompt text -> token ids + visual inputs`

并且说清楚每一步由谁负责，这个里程碑才算完成。

## 里程碑 3：解剖视觉 Token 长度与视觉 Tensor 路径

### 为什么这一步必须单独拿出来

因为这是多模态项目里最容易“看懂现象，但没有真正理解”的地方。

例如你已经遇到过这个问题：

- 为什么我看到的是一串重复的 `image_token_id`？
- 这些 token 的个数到底怎么来的？
- 是按分辨率决定的吗？

如果这一步没有彻底讲明白，你后面写本地 processor 或 engine 时，长度统计和占位协议就非常容易错。

### 这里会遇到的名词

`patch`
表示图像被切出来的小块。

`merge_size`
表示若干 patch 在更高层被合并时使用的合并尺度。

`image_grid_thw`
表示图像最终映射成的视觉网格大小。

`image token count`
表示在最终 `input_ids` 中，`image_token_id` 实际重复了多少次。

### 这一步具体要做什么

你要把下面四件事串起来：

1. 模型 config 里的：
   - `patch_size`
   - `spatial_merge_size`
   - `image_token_id`
2. processor 输出里的：
   - `image_grid_thw`
3. token 序列里的：
   - `vision_start_token_id`
   - 多个 `image_token_id`
   - `vision_end_token_id`
4. 模型侧实际要求的：
   - `pixel_values`
   - `image_grid_thw`
   - image features 数量与 placeholder token 数量一致

你要真正理解的是：

- 图像 token 的个数不是凭空来的
- 它和图像预处理后的视觉网格有关
- 它最终要和模型实际消费的 image features 数量严格对齐

### 这一步先不要做什么

- 不要把“能看到重复 token”误当成已经理解了长度契约
- 不要只记住几个数字而不理解这些数字之间的关系

### 建议产出物

- 一份“视觉长度计算说明”
- 对固定样例写清：
  - `image_grid_thw = ?`
  - `merge_size = ?`
  - `image_token` 重复次数 = ?
  - 为什么是这个数

### 关键提问

- 为什么 `image_token_id` 的数值和它重复的次数是两回事？
- 为什么单张图通常不是“一个图像对应一个 token”？
- `image_grid_thw` 描述的是原图尺寸吗？如果不是，它描述的是什么？
- 为什么 `image_token` 的数量必须和模型侧 image features 的数量对齐？
- 如果本地框架长度统计错了，会优先表现成哪类 bug？
- 为什么多模态里“能输出一些 token”和“真的理解图像”不是同一件事？

### 完成标准

当你能拿着一个具体样例，把“图像 -> patch/grid -> image token count -> model inputs”这一整条链讲清楚，这个里程碑才算完成。

## 里程碑 4：建立本地教学抽象与模块边界

### 为什么到这一步才开始谈设计

因为前 3 个里程碑解决的是“真实世界到底怎么工作”的问题。
到了这里，你才有资格回答“本地框架应该怎么组织代码”。

如果这一步提前做，很容易变成：

- 先拍脑袋定义几个类
- 再被官方真实契约反复打脸

### 这里会遇到的名词

`abstraction`
就是抽象。
在这里指的是：把复杂真实系统整理成几个职责清楚、便于学习的小模块。

`boundary`
就是边界。
指的是：一个模块应该知道什么，不应该知道什么。

### 这一步具体要做什么

基于当前仓库，给自己定义一套最小教学抽象。
推荐至少包含：

- `VLRequest`
  - 表示上游请求
  - 应该更接近 `messages`，而不是 `prompt + image_path`
- `VLProcessorOutput`
  - 表示 processor 的输出契约
  - 应该能容纳 prompt 侧和视觉侧信息
- `VLProcessor`
  - 负责输入装配
- `VLModelAdapter`
  - 负责框架与真实 checkpoint 对接
- `VLEnginePath`
  - 负责 prefill/decode、长度统计、输出裁剪

这一步不要求你已经实现所有类。
要求的是你先写清楚它们各自的职责。

### 这一步最重要的判断

你要明确区分：

- 哪些职责属于 processor
- 哪些职责属于 adapter
- 哪些职责属于 engine

例如：

- engine 不应该拥有“图像 token 如何展开”的知识
- adapter 不应该拥有“上游 `messages` 怎么构造”的知识

### 这一步先不要做什么

- 不要提前把所有通用化接口做大
- 不要为了“以后可能支持更多模型”而先做复杂抽象

### 建议产出物

- 一份小型设计说明
- 一个职责边界表

### 关键提问

- 为什么现在适合先做“最小抽象”，而不是“通用抽象”？
- 本地 engine 最好知道哪些多模态信息，最好不知道哪些信息？
- 为什么 `VLModelAdapter` 是这个阶段的关键，而不是“自己重写完整模型”？
- 如果一个模块同时知道 prompt 规则、图像预处理和 decode 细节，它大概率出了什么设计问题？
- 现有仓库里的哪些模块可以保留，哪些模块只适合作为学习遗产？

### 完成标准

如果你能把“输入装配、模型对接、推理执行”三部分清晰拆开，并且说出每个部分的职责边界，这个里程碑才算完成。

## 里程碑 5：实现教学版 `VLProcessor`

### 为什么先实现 Processor，而不是先改 Engine

因为在真实 VL 路径里，engine 解决的是“怎么跑”的问题。
而 processor 解决的是“到底把什么送进去”的问题。

输入都没稳定，engine 层改得再多，也只是加快错误传播。

### 这里会遇到的名词

`teaching version`
表示“教学版实现”。
它不追求覆盖官方全部参数和边界情况，而追求：

- 结构清楚
- 能跑最小样例
- 行为对齐关键真值

### 这一步具体要做什么

实现一个最小可用的本地 `VLProcessor`，它至少能：

1. 接收结构化输入
2. 生成或托管 prompt text
3. 生成 `input_ids`
4. 产出视觉侧输入：
   - `pixel_values`
   - `image_grid_thw`
5. 把这些内容组织成一个清晰的 `VLProcessorOutput`

这里有一个很重要的教学选择：

- 第一版可以适度依赖官方 processor 做底层工作
- 但你必须自己定义外层输入输出结构和职责边界

也就是说，第一版可以是“自己包装 + 对齐官方”，不必一开始就“自己重写所有图像预处理细节”。

### 这一步先不要做什么

- 不要为了证明掌握而硬写一份完整官方 processor 复刻版
- 不要一开始就把 Processor 写成支持所有模型的通用超类

### 建议产出物

- 一个教学版 `VLProcessor`
- 一组与官方基线并排输出的对照日志

### 关键提问

- 本地 `VLProcessor` 相比官方 processor，哪些地方是“教学抽象”，哪些地方必须严格对齐？
- 为什么“允许第一版依赖官方 processor 的底层能力”并不等于“只是抄官方”？
- `VLProcessorOutput` 至少应该包含哪些字段？为什么不能只返回 `input_ids`？
- 对固定样例，本地 `VLProcessor` 的输出与官方输出，哪些字段必须一致，哪些字段允许实现方式不同？
- 如果你把 `messages` 在最外层就压扁成 `prompt: str`，后面会失去什么？

### 完成标准

如果你能对同一个样例并排展示：

- 官方 processor 输出
- 本地教学版 `VLProcessor` 输出

并说明它们在关键字段上如何对齐，这个里程碑才算完成。

---

## 里程碑 6：实现真实 Checkpoint `VLModelAdapter`

### 为什么不是“手写一个完整 VL 模型”

因为那不是当前最有学习收益的路径。

你现在最需要弄明白的是：

- 本地框架怎样和真实 Qwen2.5-VL checkpoint 对接

而不是：

- 先把 Qwen2.5-VL 内部结构一层不差地重新造一遍

`VLModelAdapter` 是一个高杠杆模块。
它能让你先把“真实模型语义”接进来，再判断本地 engine 的哪些抽象成立，哪些不成立。

### 这里会遇到的名词

`logits`
表示模型对下一个 token 的原始预测分数。
如果你的本地采样器还要继续工作，它最终就需要这类输出。

`forward pass`
表示一次前向计算。
在这里通常是指把 `input_ids + pixel_values + image_grid_thw ...` 等输入送进真实模型。

### 这一步具体要做什么

实现一个最小 `VLModelAdapter`，至少回答这些问题：

- 本地 `VLProcessorOutput` 如何映射到真实模型的输入字段？
- 真实模型返回哪些结果？
- 本地代码需要消费哪些结果？
- 在最小样例里，本地路径是否能成功完成一次真实 forward / generate？

第一版允许你直接依赖 Hugging Face 的真实 Qwen2.5-VL 模型。
重点不是去掉外部依赖，而是把本地的 adapter 边界搭对。

### 这一步先不要做什么

- 不要急着重写视觉塔
- 不要急着自己接管所有 position ids 和 RoPE 细节
- 不要急着做性能优化

### 建议产出物

- 一个最小 `VLModelAdapter`
- 一条可以调用真实 checkpoint 的本地推理路径

### 关键提问

- 为什么 adapter 比“完整模型重写”更适合作为当前阶段的目标？
- 本地框架交给 adapter 的应该是什么，而不是直接把所有模型细节散落到 engine 里？
- 为什么说“模型能 forward”只是中间里程碑，不等于“VL 系统已经完成”？
- 对单个样例，adapter 层最少要保证哪些字段和语义对齐？
- 如果模型输入 keys 对了，但结果仍然很怪，优先该回头检查哪几个环节？

### 完成标准

如果本地路径已经能利用 `VLProcessorOutput` 成功调用真实 Qwen2.5-VL checkpoint，并获得结构合理的输出，这个里程碑才算完成。

---

## 里程碑 7：把真实路径接入 mini-vLLM 引擎并验证 Prefill / Decode

### 为什么这是最后一步，而不是前面的起点

因为这里讨论的是“服务框架语义”。
它比“输入契约”和“模型对接”更靠后。

如果你前面的理解还不稳，就会把所有问题都误诊成 engine bug。
这在多模态项目里非常常见。

### 这里会遇到的名词

`KV cache`
表示模型为了高效增量生成而缓存的 attention 中间状态。

`length accounting`
表示框架如何统计序列长度。
在多模态里，这尤其重要，因为图像 token 也占据上下文位置。

### 这一步具体要做什么

把前面建立好的真实路径接进当前学习型引擎，重点验证：

1. prefill 阶段是否正确包含了视觉前缀
2. decode 阶段是否只追加文本 token
3. 本地长度统计是否与真实 placeholder token 数量一致
4. 本地 sampler 是否仍能消费真实模型输出
5. prompt trimming 与 completion 语义是否仍然成立

这一阶段最重要的不是“高性能”，而是“语义自洽”。

### 这一步先不要做什么

- 不要把所有奇怪输出都归因于 scheduler
- 不要用“调大 `max_model_length` 后能跑更久”当作已经成功的证据
- 不要急着追求性能 benchmark

### 建议产出物

- 一条贯通的真实 VL 推理路径
- 一份“引擎假设检查表”

### 关键提问

- 为什么图像通常只参与 prefill，而不是每一步 decode 都重新参与？
- 本地框架的长度统计里，图像 token 应该如何被计入？
- 如果 prefill/decode 契约不一致，最常见会出现什么症状？
- 哪些当前引擎抽象仍然成立，哪些必须局部修改？
- 为什么“能生成文本”不等于“视觉条件已经真正生效”？

### 完成标准

如果你能清楚说明：

- 真实 VL 路径是如何进入当前 engine 的
- 哪些假设被保留
- 哪些假设被修正
- 哪些问题仍未解决

并且对单图单轮样例已经跑通一致的推理流程，这个里程碑才算完成。

---

## 建议的阶段性停顿点

为了避免一口气写太多代码，建议在下面三个点停下来复盘。

### 停顿点 A：完成里程碑 3 之后

此时你应该已经不再把多模态理解成：

- “文本 prompt 外加一点额外 embedding”

而应该开始把它理解成：

- “结构化输入 + 官方 prompt 协议 + 视觉 tensor + 对齐的 token 长度”

如果这一步还没稳，不要进入本地抽象设计。

### 停顿点 B：完成里程碑 4 之后

此时你应该能不用写实现，就先说清楚：

- processor、adapter、engine 各自应该负责什么

如果这一点还说不清，后面的代码大概率会互相污染职责。

### 停顿点 C：完成里程碑 6 之后

此时你应该已经证明：

- 本地教学路径不是“自娱自乐”
- 它已经能和真实 checkpoint 对接

如果这个点成立，再去接 engine 就会稳很多。

---

## 明确不建议现在做的事

下面这些事不是永远不做，而是**不适合当前教学路线的第一阶段**：

- 同时支持多个 VL 模型
- 一开始就重写完整官方 processor
- 一开始就重写完整视觉塔
- 先做高性能优化再补理解
- 同时做视频支持
- 同时做统一多模态平台

原因不是它们不重要，而是它们会稀释这条路线最核心的学习收益：

- 理解真实 Qwen2.5-VL inference contract
- 把它翻译成一个教学型 mini framework

## 最终完成定义

如果这个项目第一阶段完成，你应该能够做到下面几件事。

### 你能讲清楚

- 官方 Qwen2.5-VL 的最小输入契约是什么
- processor 为什么不能被简化成 tokenizer 包装器
- 图像 token 长度为什么是那个数
- 为什么 `pixel_values`、`image_grid_thw` 和 placeholder token 数量必须对齐
- prefill / decode 在多模态场景下各自承担什么职责

### 你能展示代码

- 一个教学版 `VLProcessor`
- 一个真实 checkpoint `VLModelAdapter`
- 一条接入当前 mini-vLLM 引擎的真实 VL 路径

### 你能证明没有学偏

- 你的关键样例可以和官方基线逐项对照
- 你的本地实现虽然更小、更清楚，但没有脱离官方可观测契约

如果这三类事情都能做到，这个项目就已经有很强的学习含金量了。

## 最后的判断标准

决定要不要继续做这条路线，最好的判断方式不是问：

- “这个项目最终能不能替代 vLLM？”

而是问：

- “如果我完成这份路线图，我是否会真正理解一个真实 VL 模型是如何从结构化输入走到最终推理输出的？”

如果答案是“会”，那这条路线就是值得投入的。

因为它交付的不是一个仓促堆出来的 demo，而是一套真正可迁移的理解：

- 你以后看 vLLM、看 Transformers、看别的 VL serving 实现时，都会更容易抓到真正重要的边界。
