# Skills — Documentation Conventions for MinivLLM-Learn

## 文档目标

本文件用于约定本仓库以后如何编写、命名、放置和维护文档。

它只关注本项目自己的文档问题：

- 文档应该放在哪里
- 文档文件名怎么取
- 文档应该写什么、不写什么
- 文档更新时如何避免堆积历史垃圾

不追求做成通用文档规范大全，也不照搬其他项目的 agent/skills 体系。

---

## 目录与路径约定

### 1. skills 类文档固定放在 `skills/`

所有“长期有效的约定类文档”放在：

- `skills/`

这类文档的特点是：

- 不对应某一次具体实现
- 是以后反复会用到的规则
- 面向后续 agent / LLM / 人类协作者

例如：

- `skills/skill-documentation.md`
- `skills/skill-code-change.md`

### 2. 总结类 / 分析类 / 学习类文档固定放在 `Docs/`

所有“针对某次分析、某次学习、某个阶段总结”的 Markdown，统一放在：

- `Docs/`

例如：

- 某个模块源码分析
- 某次 milestone 总结
- 某个外部项目实现路线归纳
- 调试过程整理后的结论文档

不建议再散落在仓库根目录。

### 3. 根目录只保留少量高层文档

根目录优先只保留：

- `README.md`
- `README_zh.md`
- `agent.md`
- 少量确实属于项目入口级别的文档

如果某个文档更像“过程记录 / 学习笔记 / 专题分析”，优先放到 `Docs/`。

## 文件命名约定

### 1. `skills` 中文档命名格式

统一使用：

- `skill-xxx.md`

其中 `xxx` 必须体现用途，而不是宽泛名字。

推荐：

- `skill-documentation.md`
- `skill-code-change.md`
- `skill-multimodal-debugging.md`

不推荐：

- `skill.md`
- `rules.md`
- `notes.md`

### 2. `Docs` 中文档命名要求

`Docs/` 下的文件名要：

- 简洁
- 能直接说明主题
- 尽量避免空泛词
- 尽量不用“new / final / tmp / test”之类无信息量后缀

推荐：

- `vllm-multimodal-analysis.md`
- `milestone1-mm-prefill-notes.md`
- `qwen3-runner-callchain.md`

可接受：

- `vLLM_multimodal_analysis_for_myvllm.md`

不推荐：

- `notes1.md`
- `summary-final-new.md`
- `some_thoughts.md`

### 3. 文件名尽量对应“一个主题”

一个文档只承载一个主问题。

不要把这些混在同一个文件里：

- 一半是源码分析
- 一半是 bug 排查日志
- 一半是 TODO

如果主题变了，应该新建文件而不是无限续写。

## 文档类型与职责边界

### 1. `skills/`：长期规则

这类文档回答的是：

- 以后应该怎么做
- 以后应该遵守什么边界

它不是：

- 一次性任务总结
- 临时调试日志
- 具体功能说明书

### 2. `Docs/`：阶段性知识沉淀

这类文档回答的是：

- 这次我们分析出了什么
- 当前代码链路是什么
- 当前 milestone 的设计与问题是什么

它应该是整理后的结果，不应该是原始聊天记录。

### 3. `agent.md`：当前阶段作战清单

`agent.md` 只保留：

- 当前任务必须知道的系统现状
- 当前阶段的 TODO / 目标 / 边界

`agent.md` 不应该替代：

- `Docs/` 的长期分析文档
- `skills/` 的长期规范文档

## 文档内容要求

### 1. 先写结论，再写展开

多数文档应优先使用这种结构：

1. 先说结论
2. 再给背景
3. 再讲代码位置
4. 再讲细节和例子

这样新读者可以先建立总览，再决定是否继续看细节。

### 2. 结合真实代码位置

只要文档讨论的是仓库里的实现，就尽量带上：

- 文件路径
- 关键类名
- 关键函数名

例如：

- `src/myvllm/engine/model_runner.py`
- `MMModelRunner.prepare_prefill()`
- `Qwen3Model.forward()`

避免整篇只有抽象概念，没有落点。

### 3. 强调“当前版本”的真实情况

文档里应尽量区分：

- 已经实现
- 当前建议
- 后续可选
- 暂未实现

不要把“计划做的内容”写得像“已经存在的实现”。

### 4. 允许写面向学习的解释，但要服务于代码理解

本项目是学习项目，所以允许写：

- 通俗解释
- 类比
- 调用链拆解
- 为什么这么设计

但这些解释必须服务于理解代码，不要变成空泛科普。

## 文档更新原则

### 1. 优先覆盖过期结论，不要无限叠加

如果旧文档结论已经不适用：

- 直接改写
- 或新建替代文档并说明旧文档已过期

不要把多个相互冲突版本长期堆在一起。

### 2. 调试日志要提炼后再落文档

不要把一长串“试了什么命令、报了什么错”的原始过程直接当正式文档保存。

更好的方式：

- 保留最终问题
- 保留根因
- 保留修复思路
- 保留关键验证点

### 3. 一篇文档完成后，至少回答这 3 个问题

- 它解决了什么认知问题？
- 它对应哪些关键代码位置？
- 它对后续开发有什么直接帮助？

如果答不出来，说明文档大概率还不够聚焦。

## 推荐模板

### A. 分析类文档

推荐结构：

```markdown
# 标题

## 先说结论

## 背景与目标

## 当前实现位置

## 关键调用链

## 核心问题或设计点

## 建议 / 下一步
```

### B. milestone / 总结类文档

推荐结构：

```markdown
# 标题

## 目标

## 当前状态

## 关键改动

## 遇到的问题与根因

## 当前结论

## 下一步建议
```

---

## 禁止事项

以后写文档时，尽量避免这些问题：

1. 文档散落在根目录，找不到主题归属
2. 标题和文件名都过于空泛
3. 结论、计划、现状混在一起不区分
4. 没有具体代码位置支撑
5. 把聊天记录原样堆成文档
6. 旧结论明显过期却还继续误导后续工作

---

## 一句话原则

> `skills/` 放长期规则，`Docs/` 放主题明确的总结与分析；每篇文档都要让后来的人能快速知道“这篇在解释什么、代码在哪、接下来能怎么用”。
