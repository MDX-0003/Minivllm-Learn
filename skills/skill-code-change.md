# Skills — Code Change Conventions for MinivLLM-Learn

## 文档目标

本文件用于约定本仓库以后进行代码修改时的基本原则。

它主要服务于这个项目当前的实际情况：

- 一个用于学习和拆解 vLLM / Qwen3 的小型仓库
- 代码量不算大，但核心路径清晰
- 需要尽量保住 text-only 主链的稳定性
- 新功能通常以“边学边做”的方式递进开发

所以这里强调的是：

- 小步可验证
- 尽量少破坏旧路径
- 每次改动要有清晰边界
- 优先为后续理解服务

---

## 总体原则

### 1. 先理解链路，再改代码

改之前优先回答：

- 当前入口在哪里？
- 当前数据结构怎么流转？
- 这个变量/函数会被哪些地方依赖？
- 我要改的是“局部实现”还是“跨层契约”？

如果还没搞清调用链，就不要一上来大改。

### 2. 优先做“增量扩展”，少做“大重写”

在这个仓库里，优先顺序是：

1. 新增类 / 新增模块
2. 在旧代码上增加最小入口
3. 只有在完全没有入口时，才直接改核心基类

例如：

- `Sequence` -> `ImageSequence`
- `ModelRunner` -> `MMModelRunner`
- `Qwen3ForCausalLM` -> `MMQwen3ForCausalLM`

这样可以最大程度保住原有 text-only 路径。

### 3. 保持 text-only 主链尽量不受影响

本项目很多代码是用来学习 text-only 推理引擎主链的。

所以新增多模态、调试逻辑、实验逻辑时，应尽量满足：

- 默认 text-only 行为不变
- decode 快路径尽量不变
- 未启用的新功能不要污染老路径

### 4.对于新增的关键代码要留有简要注释
对于用户来说新增的代码可能过于复杂，逻辑一时难以接受。
所以新增的代码最好说清楚关键变量的用意、函数的功能，torch代码里往往会出现多个class里不同函数混合调用的情况
这种情况也最好说清楚“某个返回值在外部用来做什么”、“某个输入参数是从什么地方得来的”
除了在注释里写上“做了什么”，“为什么要做”也很重要
---

## 改动边界约定

### 1. 每次改动最好对应一个“小闭环”

一个小闭环通常指：

- 一个独立概念
- 一个明确文件范围
- 一个可验证结果

例如：

- 给 `Qwen3Model.forward()` 增加 `inputs_embeds` 支持
- 给 `ImageSequence` 增加 `num_vision_tokens`
- 给 `MMModelRunner.prepare_prefill()` 修正 `slot_mapping`

不推荐把下面内容揉成一个 patch：

- 改序列结构
- 改 runner
- 改模型
- 顺手再重构 scheduler
- 再顺手改 benchmark

### 2. 每一步都要有“本步不做的事”意识

即使没专门写文档，也要在实现时明确：

- 这一步只解决什么
- 这一步明确不解决什么

例如：

- 这一步只打通 fake vision
- 不做真实 vision tower
- 不兼容官方 VL 权重
- 不动 decode CUDA Graph

### 3. 避免顺手重构无关部分

如果不是当前问题直接需要，不要因为：

- “顺眼”
- “看起来可以更优雅”
- “顺便统一命名”

就重构无关逻辑。

本项目现阶段更需要：

- 清晰可追踪
- 易验证
- 易回退

而不是“大面积变漂亮”。

## 与当前项目强相关的规则

### 1. 涉及多模态时，优先保护 decode 路径

如果一个改动会碰到：

- `prepare_decode()`
- `run_model(..., is_prefill=False)`
- CUDA graph capture

应优先问自己：

- 这个改动是不是只该发生在 prefill？
- decode 有没有必要知道图片？
- 能不能把影响限制在 prefill？

原则上：

- 视觉信息优先只进入 prefill
- decode 尽量继续只处理新 token + KV-cache

### 2. 涉及 KV-cache / `slot_mapping` 时，先对齐长度契约

多模态开发中，最容易炸的不是“模型答得不对”，而是：

- token 数不一致
- `slot_mapping` 长度不一致
- `context_lens` 不一致
- `last_block_num_tokens` 算错

所以只要改到了下面任意一处：

- `num_tokens`
- `prepare_prefill()`
- `prepare_decode()`
- `ImageSequence`
- `block_table`

都应该优先检查：

$$
\text{模型真正处理的 token 数} = \text{KV-cache 实际写入的 token 数}
$$

### 3. 如果是“学习版功能”，可以先 fake，再 real

这个仓库允许分阶段：

- 先做 fake 版本打通链路
- 再做真实版本提升语义质量

例如：

- `fake_vision_embeds` 先打通 prefill 线路
- 后续再替换成真实 vision encoder + projector

但是 fake 版本也要满足：

- 契约正确
- 路径清晰
- 后续容易替换

不能为了临时跑通而把结构搞乱。

## 文件与模块修改建议

### 1. 新能力优先单独建文件

当新增的是一个新职责时，优先新建文件，而不是一股脑塞到旧文件里。

例如：

- 多模态输入处理 -> `src/myvllm/multimodal/processor.py`
- 视觉编码 -> `src/myvllm/multimodal/vision_encoder.py`
- 投影层 -> `src/myvllm/multimodal/projector.py`

### 2. 修改核心文件时，优先做“最小入口改动”

比如：

- `Qwen3Model.forward()` 增加 `inputs_embeds`

这类改动可以接受，因为它是新能力必须经过的入口。

但修改时应尽量：

- 保持原有参数兼容
- 不改变旧调用方行为
- 不顺手清理无关实现

### 3. 注释要解释“为什么”，不只解释“做了什么”

尤其是以下场景：

- workaround
- 兼容 placeholder tensor
- 为了不破坏 decode 而特殊处理 prefill
- 不调用父类 `__init__` 这种高风险选择

这类地方，注释应明确说明原因和约束。

---

## 验证要求

### 1. 改完后至少做一次最小验证

如果改的是代码逻辑，至少要验证和本改动最相关的一条路径。

例如：

- 改 prefill，就跑一次最小 prefill 链路
- 改多模态入口，就跑一次 `main.py`
- 改序列长度逻辑，就检查 `slot_mapping` / `num_tokens` 是否一致

### 2. 优先验证“契约”和“边界”

本项目很多 bug 不是语法错误，而是边界错了。

所以比起只看“有没有报错”，更应该看：

- 长度是否一致
- decode 是否仍走原路径
- prefill / decode 的职责是否混了
- 新字段是否真正被上下游正确使用

### 3. 如果是实验性改动，要在说明里标清限制

比如：

- 只支持单图
- 只支持 prefill 注入
- 只支持 fake vision
- 不支持 prefix cache

不要让实验性改动伪装成完整能力。

---

## 与文档联动的要求

### 1. 非平凡代码改动，应同步产出说明文档

如果一次改动涉及：

- 新调用链
- 新契约
- 多文件协作
- 容易反复踩坑的 bug

建议同步在 `Docs/` 下产出总结文档。

### 2. 文档文件名要能直接说明主题

例如：

- `Docs/mm-prefill-slot-mapping-notes.md`
- `Docs/qwen3-forward-inputs-embeds.md`

不要使用：

- `Docs/notes2.md`
- `Docs/debug-final.md`

### 3. `agent.md` 只保留当前阶段最关键的作战信息

长期规则放 `skills/`

专题总结放 `Docs/`

当前阶段执行清单放 `agent.md`

不要把三者混成一个文件。

---

## 禁止事项

以后改代码时，尽量避免这些问题：

1. 没搞清调用链就直接大改核心类
2. 为了新功能把 text-only 主链一并重构
3. 在一个 patch 里混入太多无关重构
4. 只让代码“跑起来”，却不验证长度/缓存契约
5. 把实验性 hack 写成长期正式结构却不加说明
6. 改完没有留下足够上下文，导致下次还得重新理解

---

## 推荐工作方式

每次进行中等以上复杂度改动时，推荐按这个顺序：

1. 先确认当前入口、调用链、关键数据结构
2. 明确这一步只改什么、不改什么
3. 优先通过新增类/新增文件扩展
4. 只在必要入口做最小基类改动
5. 改完先做最小验证
6. 如有必要，在 `Docs/` 留下主题明确的总结

---

## 一句话原则

> 这个仓库的代码修改，优先追求“可理解、可验证、可回退”的小步扩展，而不是一次性做大而全的重构。
